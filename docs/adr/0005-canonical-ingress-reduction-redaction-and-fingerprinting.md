# ADR-0005: Canonicalize, Reduce, Redact, and Key-Fingerprint Ingress Before Journaling

## Context

ADR-0004 requires the ingestion path to persist a validated, canonical, redacted receipt before returning `accepted`.

The persistence boundary now exists, but the project does not yet have an executable definition of:

- which source fields may enter the journal;
- how unknown source fields are handled;
- how secrets and local paths are removed or reduced;
- how semantically identical JSON is represented deterministically;
- how duplicate deliveries are fingerprinted without storing raw secrets;
- how truncation and reduction are reported without leaking content;
- how the daemon and future Hook Adapter share one security policy.

Persisting the validated Claude payload directly is not acceptable. A valid Hook payload can still contain:

- source code;
- prompts;
- command lines;
- environment values;
- authorization headers;
- access tokens;
- passwords;
- private keys;
- database connection strings;
- absolute local paths;
- large tool outputs.

The source contracts deliberately preserve unknown fields for forward compatibility. Forward-compatible parsing does not mean unknown fields are safe to persist.

The project therefore needs a pure, deterministic security boundary between runtime validation and durable journaling.

---

## Decision

OwnLoop will introduce a shared package:

```text
packages/ingress-security/
```

The package will accept only values that have already passed `ClaudeAdapterIngressSchema` and will produce a versioned, OwnLoop-owned prepared receipt contract.

The critical path becomes:

```text
JSON request body
→ ClaudeAdapterIngressSchema
→ canonical source fingerprint
→ allowlist reduction
→ deterministic secret and path redaction
→ canonical redacted JSON
→ output-size validation
→ prepared receipt
→ SQLite commit
→ accepted response
```

No caller may write an ingress receipt until this boundary succeeds.

---

## Package ownership

### `packages/contracts`

Owns runtime schemas and inferred TypeScript types for:

- `PreparedIngressReceiptV1`;
- redaction summary;
- canonicalization and redaction version constants;
- structured ingress-security errors when those errors cross a package boundary.

Contracts contain no redaction behavior.

### `packages/ingress-security`

Owns pure deterministic behavior for:

- JSON compatibility validation;
- canonical JSON generation;
- source-event identifier extraction;
- keyed source-payload fingerprinting;
- deduplication-key derivation;
- event-specific field allowlisting;
- recursive secret redaction;
- local path reduction;
- deterministic truncation and size enforcement;
- redaction summary generation.

The package may depend on:

- `@ownloop/contracts`;
- `@ownloop/event-model` when JSON value types are needed;
- Node.js built-in modules such as `node:crypto`, `node:path`, and `node:url`.

It must not depend on the daemon, persistence implementation, HTTP framework, AI provider, or UI.

### `apps/daemon/src/persistence`

Owns a new immutable migration and repository mapping needed to persist the prepared receipt.

The persistence layer still does not perform redaction.

## Prepared receipt contract

The security boundary produces a contract equivalent to:

```ts
{
  canonicalizationVersion: 1;
  redactionPolicyVersion: 1;
  ingressContractVersion: 1;
  source: "claude_code";
  adapterVersion: SemVer;
  sourceSessionId: string;
  sourceEventName: SupportedClaudeHookName;
  sourceEventId: string | null;
  canonicalWorkspacePath: string;
  receivedAt: string;
  payloadFingerprint: `hmac-sha256:${string}`;
  deduplicationKey: string;
  redactedPayloadJson: string;
  redactionSummary: RedactionSummaryV1;
}
```

The contract does not include:

- a receipt ID;
- a creation timestamp generated by OwnLoop;
- processing status;
- unredacted input;
- an HMAC key;
- a generic `payload` field.

The future HTTP application layer will add receipt ID, creation timestamp, and pending status immediately before durable insertion.

## Canonical workspace path

The raw `cwd` value is required to recover and process a receipt after a daemon restart.

OwnLoop will therefore treat the canonical workspace path as dedicated **local-sensitive routing metadata**, not as ordinary payload content.

Rules:

- it is stored in a dedicated column, never inside `redacted_payload_json`;
- it is never written to ordinary logs;
- it is never included in external AI input without later independent reduction;
- the caller provides the current platform path context;
- the path is resolved and normalized syntactically;
- filesystem realpath and Git repository-root resolution are deferred to the Git-baseline stage;
- an empty or non-absolute result is rejected;
- machine-specific default paths are not selected by the package.

A new migration will add dedicated receipt metadata columns rather than weakening the explicit `redacted_payload_json` boundary.

---

## Canonical JSON v1

OwnLoop Canonical JSON v1 is based on the deterministic principles of RFC 8785 JSON Canonicalization Scheme.

For parsed JSON-compatible values it will:

- preserve Unicode string data as-is;
- reject lone surrogate code units;
- reject `NaN`, positive or negative infinity, `BigInt`, `undefined`, functions, symbols, sparse arrays, cyclic structures, and non-plain objects;
- reject negative zero;
- serialize JSON primitives using ECMAScript JSON serialization;
- sort object property names recursively by UTF-16 code-unit order;
- preserve array element order;
- emit no insignificant whitespace;
- encode the result as UTF-8 when measuring or fingerprinting.

The project will not claim complete RFC 8785 conformance for raw JSON text because duplicate object names cannot be recovered after ordinary JSON parsing. OL-003 may later add raw-body duplicate-key rejection if required.

The canonicalizer must not normalize Unicode text to NFC, NFD, or another normalization form.

---

## Keyed fingerprinting

A plain cryptographic hash of raw ingress content can expose low-entropy secrets to offline guessing.

OwnLoop will fingerprint the canonical unredacted source payload with HMAC-SHA-256 using a local secret key.

Rules:

- the API accepts a Node.js secret `KeyObject`;
- the symmetric key must contain at least 32 bytes;
- the key is never serialized, logged, returned, or persisted by this package;
- the HMAC input is the canonical source `payload`, not the adapter `receivedAt` timestamp;
- the output format is `hmac-sha256:<lowercase hex>`;
- the raw canonical source string is held only in memory for the shortest practical time;
- error objects must never include the raw input or canonical source string.

Node.js built-in `createHmac` and secret `KeyObject` APIs are sufficient; no hashing dependency is added.

## Source event identifiers and deduplication

Source event IDs are extracted only where the Hook contract provides a stable identifier:

- `UserPromptSubmit`: `prompt_id` when present;
- `PreToolUse`: `tool_use_id`;
- `PostToolUse`: `tool_use_id`;
- `PostToolUseFailure`: `tool_use_id`;
- all other supported v1 Hooks: `null`.

`PostToolBatch` uses the keyed payload fingerprint because one batch contains multiple tool-use IDs.

The deduplication key is versioned:

```text
v1:<hook-name>:id:<source-event-id>
```

when a stable source ID exists, otherwise:

```text
v1:<hook-name>:hmac:<fingerprint-hex>
```

The source and source-session columns remain part of the database uniqueness scope.

The deduplication key contains no raw prompt, command, path, token, or source code.

---

## Allowlist reduction

Each supported Hook receives an explicit field allowlist.

Common fields are moved or reduced as follows:

- `session_id`: moved to `sourceSessionId`;
- `hook_event_name`: moved to `sourceEventName`;
- `cwd`: moved to dedicated `canonicalWorkspacePath` metadata;
- `transcript_path`: removed from the payload and represented only through deterministic path replacement where necessary;
- `prompt_id`: moved to `sourceEventId` when applicable;
- `permission_mode`, `effort`, `agent_id`, and `agent_type`: retained when present and redacted recursively;
- unknown common fields: dropped and counted.

Event-specific fields required for later normalization are retained through an explicit allowlist.

Unknown top-level and nested source fields are not persisted merely because the source schema accepted them. Unknown fields are dropped and counted in the redaction summary.

Adding a new persisted source field requires a policy-version change and tests.

---

## Secret-field redaction

Object property names are normalized for matching by lowercasing and removing separators such as `_`, `-`, `.`, and whitespace.

Version 1 contains an explicit exact-match denylist for secret-bearing names including:

- authorization;
- proxy authorization;
- cookie and set-cookie;
- password and passwd;
- secret and client secret;
- API key;
- access, refresh, and ID token;
- generic token;
- private key and SSH private key;
- credential and credentials.

Entire values under a denied field name are replaced with a deterministic redaction marker.

Names such as `max_tokens`, `token_count`, and `token_limit` are not matched merely because they contain the substring `token`.

No entropy-based secret detector is included in v1 because its false-positive and false-negative behavior would be difficult to reason about on the critical path.

---

## Secret-pattern redaction in strings

String values under non-secret field names are scanned for a bounded, versioned set of strong patterns:

- bearer and basic authorization values;
- PEM private-key blocks;
- password, token, API-key, and client-secret assignments;
- URI user-info passwords;
- strongly prefixed provider tokens supported by explicit tests.

Redaction replaces only the detected secret portion unless a private-key block or secret-bearing field requires replacing the complete value.

Patterns must be linear or bounded in behavior. Catastrophic-backtracking regular expressions are prohibited.

The rules are not presented as a complete secret scanner. Unsupported patterns remain a documented risk.

---

## Path reduction

The redactor receives explicit path context:

- canonical workspace path;
- source transcript path;
- optional home directory.

In retained string values it replaces:

- workspace-root occurrences with `$WORKSPACE`;
- the source transcript path with `$CLAUDE_TRANSCRIPT`;
- home-directory occurrences outside the workspace with `$HOME`.

For structured file-path fields:

- paths inside the workspace become `$WORKSPACE/<relative-posix-path>`;
- the transcript path becomes `$CLAUDE_TRANSCRIPT`;
- paths under the provided home directory become `$HOME/<relative-posix-path>`;
- other absolute paths are reduced to a non-reversible placeholder plus basename;
- relative paths remain relative after separator normalization.

The original absolute path must not appear in `redactedPayloadJson`.

---

## Bounded processing

Version 1 limits are fixed constants and exported for tests:

- maximum input canonical UTF-8 size: 1 MiB;
- maximum output canonical UTF-8 size: 256 KiB;
- maximum recursive depth: 32;
- maximum object properties per object: 1,024;
- maximum array items per array: 1,024;
- maximum retained string UTF-8 size: 64 KiB.

Rules:

- oversized input is rejected before durable persistence;
- secret scanning occurs before retained-string truncation;
- arrays and strings may be deterministically truncated only when the policy explicitly allows it;
- object-property overflow is rejected rather than silently dropping arbitrary keys;
- truncation markers contain no removed content;
- the summary records truncation counts and rule identifiers, not values.

---

## Redaction summary

The summary contains only safe aggregate metadata:

```ts
{
  policyVersion: 1;
  redactedFieldCount: number;
  redactedValueCount: number;
  pathReplacementCount: number;
  droppedUnknownFieldCount: number;
  truncatedValueCount: number;
  rulesApplied: string[];
  outputUtf8Bytes: number;
}
```

`rulesApplied` is sorted and unique.

The summary must not contain:

- original values;
- original paths;
- source-code excerpts;
- prompt excerpts;
- commands;
- token prefixes copied from input.

---

## Persistence migration

A new immutable migration will extend `ingress_receipts` with dedicated prepared-receipt metadata:

- `canonicalization_version`;
- `redaction_policy_version`;
- `adapter_version`;
- `canonical_workspace_path`;
- `redaction_summary_json`.

The existing migration must not be modified.

Migration tests must prove:

- upgrade from migration version 1;
- idempotent reopen;
- valid-JSON enforcement for the summary;
- repository insert/read behavior;
- no unredacted or generic raw payload column is introduced.

---

## Error policy

Ingress-security errors are typed and contain only stable metadata.

Initial error categories include:

- invalid JSON-compatible value;
- canonicalization failure;
- invalid or insufficient HMAC key;
- invalid workspace path;
- input too deep;
- input too large;
- output too large;
- unsupported source Hook;
- internal policy invariant violation.

Errors may contain a structural JSON path composed only of field names and array indexes.

Errors must not include the rejected value, raw JSON, prompt, command, source code, secret, or absolute path.

## Consequences

### Positive

- the journal receives one well-defined safe representation;
- retries can be deduplicated without storing raw secret material;
- daemon and Hook Adapter can share one policy implementation;
- unknown source fields do not become persistent data by accident;
- canonical fingerprints are stable across object insertion order;
- redaction behavior is versioned, testable, and reproducible;
- downstream processing can recover required workspace routing metadata after restart.

### Negative

- deterministic redaction cannot guarantee detection of every secret format;
- reduction may remove future source fields until the policy is deliberately updated;
- path and string processing add latency to the critical path;
- a dedicated HMAC key must later be generated and protected by local installation logic;
- migration version 2 expands the ingress journal schema;
- full RFC 8785 raw-text conformance is not claimed.

### Accepted risks

- strong-pattern and field-name redaction is accepted for v1 instead of entropy heuristics;
- local canonical workspace paths are stored because restart-safe processing requires them;
- synchronous canonicalization is accepted under strict request-size limits;
- raw source content exists transiently in process memory before redaction but is never persisted or logged.

---

## Explicitly out of scope

This decision does not authorize:

- HTTP or Fastify endpoints;
- local installation-token generation;
- stdin handling;
- Hook forwarding;
- Claude settings changes;
- receipt insertion orchestration;
- background receipt processing;
- lifecycle resolution;
- normalized event creation;
- sequence allocation;
- Git analysis;
- artifact content storage;
- external AI calls;
- UI work;
- telemetry or analytics.

---

## Validation

This decision is validated when tests prove:

1. canonical output is stable across object insertion order;
2. array order remains significant;
3. Unicode is preserved without normalization;
4. negative zero, lone surrogates, cycles, non-finite numbers, and non-JSON values are rejected;
5. field-name and strong-pattern secrets are absent from output;
6. workspace, transcript, and home paths are reduced correctly;
7. unknown fields are dropped and counted;
8. limits and deterministic truncation behave as documented;
9. fingerprints are stable for retries and differ when source payload content differs;
10. the fingerprint key is never returned or logged;
11. source-event ID extraction covers all nine supported Hooks;
12. errors contain no rejected content;
13. migration version 2 upgrades a version-1 database;
14. only prepared redacted JSON can be inserted through the repository API;
15. standard formatting, lint, typecheck, tests, and build pass.

---

## Reversibility

Redaction rules and canonical formats are versioned and can evolve through new policy versions.

Changing the meaning of an existing fingerprint, deduplication key, canonicalization version, or redaction policy version requires a new ADR and compatibility plan.
